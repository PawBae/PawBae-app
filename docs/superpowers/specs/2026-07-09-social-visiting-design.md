# PawBae 社交原型：异步串门与共同记忆 规划

日期：2026-07-09

关系：承接 [平台愿景 P0](../../strategy/2026-07-09-platform-vision.md) 的 P4 社交路线；依赖 [平台 P1 服务端地基](2026-07-09-platform-foundation-design.md) 的账号、宠物快照、隐私门与 Realtime。本文件定义产品方向与验证切片，不是实现计划。

## 0. 已确认的产品决定

| 岔路口 | 决定 |
|---|---|
| 第一次社交的核心感受 | **“我的宠物真的交到了一个朋友”**，不是完成一次关注或领取奖励 |
| 第一社交动作 | **异步串门**，而不是实时 PK、聊天或送数值礼物 |
| 宠物是否真的离家 | **真的离开**；平台上每只宠物同一时刻只有一个位置，不保留宠物分身 |
| 串门期间的 agent 连接 | **远程连线**；宠物在好友家继续映射主人的 agent 状态 |
| 主动性 | 第一次由主人发起；成为熟朋友后，宠物可以提议串门，但必须由主人确认 |
| 社交红线 | never-punish：可拒绝、可召回、不掉资源、不降关系、不制造缺席焦虑 |

## 1. 产品命题

PawBae 的好友不是用户名列表，而是**两只宠物拥有共同经历**。

异步串门把 BYOA 从单机养成变成平台能力：一只宠物的身体出现在好友桌面，仍由远处主人的真实 agent 活动驱动。好友看到的不是预写脚本 NPC，而是“朋友的 agent 的脸”正在自己家生活。这个体验同时建立三类平台资产：

1. **关系资产**：谁和谁见过、共同经历过什么；
2. **记忆资产**：只有这两只宠物共同拥有的历史；
3. **身份资产**：宠物的性格、能力与关系由长期真实活动塑造，无法复制。

这也是后续技能传承与 PK 的正确起点：先成为朋友，再从共同经历中长出能力，最后让能力在舞台上表达。

## 2. 设计原则

### 2.1 唯一存在

- 每只宠物同一时刻只有一个平台位置：`home`、`traveling` 或 `visiting(friend)`。
- 串门时主人桌面不渲染宠物本体，只留下空窝、脚印和“去 Momo 家玩了”的状态牌。
- 设置、审批跳转、紧急召回等产品功能仍可从托盘或状态牌进入；不以复制宠物来维持功能。

### 2.2 关系先于奖励

- 第一次串门的主要产出是一段共同记忆，不是金币、经验或稀有掉落。
- 不设置每日串门任务、连续串门 streak、好友排行榜或错过惩罚。
- 重复串门增加的是故事密度与关系表达，不制造必须刷的数值压力。

### 2.3 主人始终有最终控制权

- 第一次串门必须由访客主人发起、主人好友接受。
- 熟朋友可开启“自动欢迎”，但宠物自主串门仍先向主人提议。
- 双方都可随时结束访问；拒绝、取消、召回、拉黑不会造成关系或资源损失。

### 2.4 只分享可解释的衍生状态

- 来访宠物只分享经过服务端白名单验证的公开投影。
- 好友可看到宠物外观、公开展示名和 `idle / working / waiting / compacting / offline` 等状态枚举。
- 不分享任务内容、代码、会话标题、项目路径、prompt、token 明细、agent 厂商或精确工作统计。
- 公开展示名与 handle 是用户明确填写并选择公开的社交资料，走独立的长度、字符集与举报规则；本地昵称不会自动上传。Connector 活动通道仍然没有任何自由文本参数位。

## 3. v1 核心循环

### 3.1 成为好友

1. 用户通过 GitHub handle、PawBae handle 或邀请链接找到对方；
2. 一方发送好友请求，对方接受后建立双向关系；
3. 只有双向好友可以发起串门，默认不允许陌生宠物来访。

联系人导入、推荐好友、通讯录匹配和公开广场均不属于 v1。

### 3.2 发起串门

1. 主人在好友卡片点击“去 Momo 家玩”；
2. 对方收到邀请，选择接受或婉拒；邀请 24 小时未处理则自然过期；
3. 接受后，宠物背上小包离家，播放短途旅行过场；
4. 平台创建一次 **30 分钟访问租约**，记录 `started_at` 与 `ends_at`；
5. v1 同一只宠物只能发起一个访问，同一家庭同一时刻只接待一位访客。

30 分钟是验证用固定值，不开放时长选择，减少协调与边界状态。主人可随时提前召回。

### 3.3 在好友家生活

- 好友桌面同时渲染主人宠物与来访宠物；来访者保留自己的名字、皮肤和主人归属。
- 两只宠物会使用少量确定性互动：碰鼻子、并排坐、追纸飞机、一起庆祝、各自睡觉。
- 好友可以摸摸访客、拍合照、请一份免费的待客点心。
- 待客行为只生成积极动画与共同记忆素材，不改变访客的饥饿、金币、成长、背包或所有权。

### 3.4 远程映射主人的 agent

来访者的渲染信号继续来自它自己的主人：

| 主人侧状态 | 好友家中的表现 |
|---|---|
| agent idle | 与主人宠物玩耍、休息或随机 idle |
| processing / tool running | 进入工作动画；长任务可在好友家搭小帐篷 |
| waiting for approval | 叼起纸条张望；好友不能读取或处理审批 |
| compacting | 播放整理记忆/收拾云朵的表现 |
| genuine completion | 两只宠物一起庆祝，但不展示任务内容 |
| connector offline | 来访宠物在客房睡觉，显示“主人的 agent 在休息” |

高频状态通过短生命周期快照或 Realtime 通道传递，不写成永久社交事件流。

### 3.5 回家与共同记忆

访问租约到期或任一方提前结束后：

1. 来访宠物向好友告别并离开；
2. 主人桌面的空窝播放归家过场，宠物恢复 `home`；
3. 平台从本次访问的结构化事实生成一条共享记忆；
4. 两只宠物的日记都保存同一个 `shared_memory_id`，但以各自口吻渲染；
5. 双方可查看合照或记忆卡，不自动公开发布。

示例：

> Momo 陪我等主人完成了一场很长的冒险。听到完成铃时，我们一起跳了起来。

记忆只使用模板键和安全参数，例如 `played_together`、`worked_together`、`celebrated_completion`、`shared_snack`、昼夜时段与时长档位；永远不包含任务自由文本。

## 4. 共同记忆与关系成长

### 4.1 v1 关系不是亲密度进度条

v1 不向用户展示可刷的 0–100 亲密度。关系页只展示：

- 第一次见面日期；
- 串门次数；
- 最近一次共同记忆；
- 共同相册；
- 从历史自然获得的关系称号，例如“第一位云上朋友”“午后常客”。

底层可以保留离散关系阶段 `new / familiar / close` 供内容解锁，但不显示倒计时或“还差几次”，避免把友情任务化。

### 4.2 共同记忆的所有权

- 共享记忆属于两只宠物，不属于某一方单方面可改写的帖子。
- 任一方可以从自己的日记隐藏或删除展示副本；不能远程修改对方已经获得的记忆。
- 拉黑后停止新互动，历史默认留在双方私人日记；用户可以选择一键隐藏与该关系相关的全部记录。

## 5. 平台边界与核心组件

本规划只定义产品边界，具体 SQL 与 API 在后续 P4 实施 spec 中确定。

### 5.1 Friendship

- 双向好友关系与请求状态；
- 支持接受、婉拒、解除、拉黑；
- 拉黑优先于所有访问授权；
- v1 好友资料默认仅对登录用户可见，不建立公开社交图谱。

### 5.2 Visit Lease

- 一次有明确开始与结束时间的访问租约；
- 状态机：`requested → accepted → traveling → visiting → returning → completed`；
- 终止态：`declined / cancelled / expired / recalled / blocked`；
- 每个写操作带客户端幂等键，重复点击或网络重试不能创建两次访问；
- 数据库保证每只宠物最多一个未结束访问，维护唯一位置不变量。

### 5.3 Public Pet Projection

- 从主人私有宠物快照生成专供好友查看的最小公开投影；
- 只包含皮肤/名字、渲染状态枚举、是否离线与必要的版本字段；
- 不能通过放宽私有 `pets.snapshot` 的 SELECT RLS 来实现社交查看；
- 访问结束或授权撤销后，好友立即失去访客 Realtime 订阅权限。

### 5.4 Connector Liveness

- P1/P2 必须提供服务端可判断的 `connector_seen_at` 或等价 heartbeat；
- 仅凭“宠物快照上次变化时间”不能区分电脑离线与在线空闲；
- heartbeat 只证明连接存活，不携带会话内容、项目标识或工作计数；
- 主人所有 Connector 均失联后，来访宠物公开状态收敛为 `offline`。

### 5.5 Shared Memory

- 保存双方宠物、访问来源、模板键、安全参数与生成时间；
- 文案在客户端按 locale 和宠物口吻渲染，不在服务端存预渲染自由文本；
- 同一访问最多结算一条主要共同记忆，幂等重放不重复写日记。

## 6. 异常与恢复规则

### 邀请时好友离线

邀请进入待处理状态，宠物仍留在自己家。对方上线接受后才开始访问租约；过期自动取消，无负反馈。

### 访问开始后主人离线

来访宠物留在好友家并进入睡眠状态。重新连线后恢复状态映射；不延长访问时间，不补发高频动画。

### 访问开始后好友离线

宠物在平台位置上仍处于好友家；好友重新打开应用且租约未到期时可继续看到访客。到期后双方都按 `ends_at` 本地推导归家，不依赖一次必须成功送达的结束消息。

### Realtime 或服务端暂时不可用

- 客户端保留最后一个通过验证的安全快照，并标记连接不稳定；
- 访问租约到期后，主人端恢复 home、好友端移除访客，避免无限卡住；
- 重连后以服务端租约与版本号收敛，不合并出两个宠物实例。

### 任一方召回、解除好友或拉黑

访问立即进入 returning/blocked，停止向好友发送新状态。拉黑是最高优先级，不能被自动欢迎或旧邀请绕过。

## 7. 隐私、安全与 never-punish 验收线

1. 好友只能在有效访问租约内读取访客的 public projection；
2. 非好友、已解除好友、被拉黑用户和过期访问均不能订阅；
3. Connector 状态与共同记忆参数由客户端构造器和服务端 CHECK/RLS 双层拒绝自由文本、未知键和越界值；公开展示名/handle 作为独立社交资料校验；
4. 好友不能远程审批、唤起工具、修改宠物状态或访问主人设备；
5. 摸摸、点心、合照不会消耗任何一方资源；
6. 拒绝邀请、提前召回、离线、错过访问都不降关系、不丢记忆、不显示责备文案；
7. 提供静音邀请、仅好友可访、逐好友自动欢迎、解除与拉黑入口；
8. 服务端限制邀请与访问频率，防止通知轰炸和配额滥用。

## 8. v1 明确不做

- 宠物聊天、主人聊天、自由文本信件；
- 传输真实 memory、prompt、`SKILL.md`、代码或可执行工具；
- 亲密度排行榜、每日串门任务、访问 streak；
- 一次接待多只访客、公共房间、广场与陌生人匹配；
- 共享资源、偷取、恶作剧、负面状态与 PvP；
- 实时语音、实时共同操控或强制双方同时在线；
- 自动公开共同记忆到主页或社交媒体。

## 9. 验证指标与阶段门槛

第一版不是验证“用户会不会点按钮”，而是验证“共同经历是否让用户想再次见面”。

### 核心漏斗

1. 好友请求发送 → 接受；
2. 好友建立 → 首次串门邀请；
3. 串门邀请 → 完成一次访问；
4. 完成访问 → 查看共同记忆；
5. 首次访问 → 7 天内再次与同一好友串门。

### 暂定验证门槛

- 好友邀请接受率 ≥ 40%；
- 已成为好友的用户中，首次串门发起率 ≥ 35%；
- 已接受访问的完成率 ≥ 70%；
- 首次访问用户中，7 天内同一关系的第二次串门率 ≥ 25%；
- 拉黑/骚扰反馈低于访问用户的 1%。

这些是封闭 beta 的决策门槛，不是长期增长 KPI。若用户只完成一次访问而不复访，优先修共同经历的情感质量，不追加更多奖励。

### 定性验证

访谈重点只问：

- “什么时候你觉得那真的是朋友的宠物？”
- “宠物回家后，你有没有想再看看对方？”
- “哪一步像普通社交 App，而不像两只宠物在交朋友？”

## 10. 分阶段落地

### P4-A：关系入口

- handle / 邀请链接；
- 双向好友请求；
- 好友列表、解除、静音与拉黑；
- 好友只能看到最小静态宠物卡。

### P4-B：串门纵向切片

- 单访客、固定 30 分钟、首次需接受；
- 唯一位置 + 访问租约；
- 好友桌面双宠渲染；
- 远程 agent 状态映射；
- 到期/召回/断网恢复。

这是第一个必须交给真实好友对测试的 aha milestone。

### P4-C：共同记忆

- 结构化访问摘要；
- 双方日记写入与共同相册；
- 关系称号；
- 熟朋友的自动欢迎与宠物提议串门。

只有 P4-B 证明用户愿意复访后才进入 P4-C 的丰富化。

## 11. 向技能传承与 PK 的演化

### 11.1 记忆种子

多次共同经历可以凝结成一颗“记忆种子”，它是结构化关系纪念物，不包含任何真实工作内容。例如：

- 一起经历多次长任务 → “长途同伴”；
- 经常一起等待审批 → “耐心守候”；
- 多次共同庆祝完工 → “击掌默契”。

种子先用于文案、动画与合照姿势，不立即提供数值优势。

### 11.2 技能传承

关系被证明有价值后，记忆种子可解锁安全的宠物技能卡。技能卡只传递平台定义的能力枚举、等级与来源关系，例如“协作鼓舞”“轮流守夜”，绝不复制真实 `SKILL.md`、prompt、memory 文件或可执行代码。

### 11.3 关系型 PK

PK 最后出现，且是关系资产的表达舞台：

- 宠物根据自己的养成与共同技能自动出招；
- 好友可以组队，也可以友谊赛；
- 输了不掉资源、不降关系、不产生失败羞辱；
- 战报讲述两只宠物如何配合或过招，而不是公开比较主人工作量。

是否进入 PK 设计，以串门复访和共同记忆保留率为门槛；社交关系没有被验证前不建战斗系统。

## 12. 测试与验收地图

### 纯逻辑

- 访问状态机所有合法/非法转换；
- 30 分钟租约、24 小时邀请过期、召回与重复请求幂等；
- 每只宠物唯一活动访问不变量；
- 公开投影 sanitizer 与未知字段拒绝；
- 共同记忆模板选择、双方同源、重复结算不重复写入。

### 服务端安全

- Alice/Bob/陌生人/被拉黑用户的完整 RLS 矩阵；
- 访问前、访问中、访问后 Realtime 授权变化；
- 伪造 pet owner、host、租约结束时间与 shared memory participants 全部拒绝；
- events、visits 与 memories 的更新/删除权限符合所有权规则；
- 邀请和访问限速无法通过换客户端绕过。

### 集成恢复

- 主人或好友在 requested/traveling/visiting/returning 各阶段断网；
- 应用关闭后跨租约结束时间再启动；
- 双端同时召回/拉黑/到期的竞态；
- 重连后不出现分身、不重复记忆、不永久留在访问状态。

### 体验验收

- 两位真实用户从互加好友到完成串门，无开发工具介入；
- 好友明确能分辨主人宠物与访客宠物；
- 访客的工作/等待/离线状态可感知但不泄露内容；
- 主人始终能找到宠物去向并一键召回；
- 共同记忆在中英文下都像宠物回忆，不像审计日志。

## 13. 商业与增长边界

串门天然形成邀请增长环：分享好友链接 → 对方领养宠物 → 第一次来访 → 生成只属于双方的共同记忆。增长素材来自真实关系，而不是平台送币。

未来可售卖旅行箱、客房主题、双宠动作、相框和记忆册封面等纯化妆品；**加好友、串门、共同记忆和基础召回永远不付费**。不出售更高亲密度、更强关系技能或更长访问时间，避免把友情变成付费数值。

## 14. 主要风险

| 风险 | 预防 |
|---|---|
| 主人因宠物离家而失去核心监控感 | 空窝状态牌保留必要入口；固定短租约；随时召回；封测观察召回率 |
| 好友只看到第二个 NPC，没有感到真人存在 | 来访者保留身份；远程映射真实 agent 状态；共同完成瞬间进入双方记忆 |
| 社交变成刷次数 | 隐藏亲密度进度；无每日任务/streak；关系内容按经历种类解锁而非次数领奖 |
| 访客骚扰或通知轰炸 | 首访接受、自动欢迎逐好友开启、限速、静音与拉黑 |
| Realtime 断线造成分身或宠物丢失 | 有期限租约、唯一活动访问约束、版本号、幂等操作与本地到期推导 |
| “分享状态”逐步滑向泄露工作信息 | 独立 public projection；固定枚举；双层白名单；不开放自由文本参数位 |
| 过早扩张到技能/PK | 以同一好友的 7 日复访为硬门；P4-B 未验证前不进入后续系统 |

## 15. 一句话验收

两位好友无需同时持续在线：一只宠物真正离开自己的桌面，出现在好友家，并继续由远处主人的 agent 状态驱动；回家时双方都获得一段不含工作内容的共同记忆。整个过程可拒绝、可召回、无损失、无分身，且让用户想让两只宠物再次见面。
