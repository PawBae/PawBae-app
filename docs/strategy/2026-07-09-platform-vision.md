# PawBae 平台愿景：宠物 Agent 平台（v1 定调）

日期：2026-07-09（与 Yining 自由脑暴后定稿）
关系：本文是 [2026-07-07 创业战略](2026-07-07-startup-strategy.md) 的增量——把阶段二尾项「小队与排行榜」的服务端基建升维为平台地基；阶段三「宠物成为界面」的路线不变。

## 1. 愿景

用户把自己的 coding agent（Claude Code / Codex / Cursor）接入 PawBae，agent 变成一只宠物；平台让用户看到彼此的宠物、聊天、投喂、PK——**QQ 宠物的 agent 版本**。产品品质对标 moneycoach.ai（独立开发者做出的苹果生态精品；他们也为 agent 开了 MCP 接入，"让用户的 agent 连进我的产品"正在成为标准动作，PawBae 做它的社交版原生玩家）。

### 核心命题：BYOA（Bring Your Own Agent）

- QQ 宠物的宠物是脚本；PawBae 的宠物背后是真 LLM，而且是**用户自己接入、自己付费的 agent**。
- 宠物人格由主人的真实工作喂养（diary 的结构化事件流已存在）→ 每只宠物是**不可复制的身份**——社交平台最贵的资产。
- 平台不出推理成本：宠物聊天/信件由用户自己的 agent 生成。别人做 AI NPC 是烧钱，我们的"NPC"是用户自带的——这是其他 AI 社交产品做不到的经济结构。

## 2. 架构定调：传感器 / 中转站 / 主舞台

```
电脑（agent 在这里跑）          云端                手机
┌──────────────────┐      ┌──────────┐      ┌──────────────┐
│ hooks / JSONL 监听 │ ───→ │ 账号+事件流 │ ───→ │ 宠物+社交+图鉴 │
│ = 传感器（已建成）  │  上传  │ = 中转站   │  推送 │ = 主舞台      │
└──────────────────┘      └──────────┘      └──────────────┘
```

- agent 是电脑上的本地进程，事件采集端**物理上只能在电脑上**（不可绕过）。
- 桌面 App 从「产品本体」转型为「最强接入端」：三家 agent 的 hook 安装、会话状态机、跨平台坑位处理就是现成的 **PawBae Connector**；未来可抽出无宠物 UI 的轻量版（menubar 图标 / `pawbae connect` CLI）。
- 手机 = 走互联网的远程舞台：与 OBS 舞台窗同一哲学——**一边是唯一大脑，另一边只管渲染**（单向快照镜像）。
- 电脑关机 ≠ 宠物死：云端存最后状态 + 异步事件（日记、问候、纪念品），宠物照常活着，头顶状态变成「主人的 agent 在睡觉」；电脑开机后事件补传。
- 云端 agent（Codex cloud / Claude Code web）未来可直连服务端、无需电脑——预留的路，v1 不做。

## 3. 两条红线

1. **隐私（生死线）**：开发者的 agent 活动涉及雇主代码。只上传衍生数值（次数 / 状态枚举 / 稀有度），**永不上传代码内容或任何自由文本**；逐项 opt-in、默认全关。事件字典白名单在 shared 包定义，客户端只能构造字典内事件，服务端再验一遍（双层门）。
2. **never-punish 延伸到社交**：PK 输了不掉任何东西；数值与工作量松耦合（工作喂养成，养成打 PK），拒绝卷王排行榜文化——云朵不责备人，云上也不内卷。

## 4. 基建决策（2026-07-09 四岔路口，全部选定）

| 岔路口 | 决定 | 理由 |
|---|---|---|
| 后端 | **Supabase** | Postgres + Auth + Realtime + Edge Functions(TS) 开箱即用；免费档够跑到几千用户；RLS 正好做隐私门；底层是开源 Postgres，可自托管逃生 |
| 账号 | **GitHub OAuth 首发** | 目标用户零摩擦，头像/用户名自带，天然适配 README 宠物牌；上 App Store 时按苹果规则补 Sign in with Apple；邮箱登录留给阶段三大众化 |
| 手机框架 | **Tauri 2 移动端** | 与桌面同代码库：精灵渲染 / 气泡组件 / 镜像协议直接复用；远程推送靠社区插件（APNs/FCM），P3 期验证 |
| 仓库 | **单仓库 workspace** | apps/desktop + apps/mobile + packages/shared + supabase/；事件 schema 一处定义、三端同步 |

## 5. 路线图（每项独立 spec→PR 循环）

- **P0** 本文档
- **P1** 服务端地基：单仓库改造（先行、零行为变化）+ Supabase 账号 / 宠物状态存储 / 事件接收 / 隐私门 → spec: [2026-07-09-platform-foundation-design.md](../superpowers/specs/2026-07-09-platform-foundation-design.md)
- **P2** 桌面 Connector：设置页「账号」区，GitHub 登录 + opt-in 上传衍生事件（默认关）
- **P3** 手机 App v1：登录即看到自己的宠物**活着**（aha moment：电脑在干活，人在咖啡店看宠物替自己盯着）
- **P4** 异步串门与共同记忆（原「三件套」升级，投喂融入待客点心）：P4-A 关系入口 / P4-B 串门纵向切片 / P4-C 共同记忆 → spec: [2026-07-09-social-visiting-design.md](../superpowers/specs/2026-07-09-social-visiting-design.md)
- **之后（按反馈排队）**：宠物信件（BYOA 生成）、挂机社交、PK（回合制、招式由真实工作衍生、输不掉东西）、世界 Boss、小队周任务、排行榜、广场/自习室、README 活体宠物牌（SVG）、宠物主页 pawbae.app/@handle、合孵/串种、观众领养主播后代蛋、conference 模式

## 6. 与现有战略的关系

- 「小队与排行榜」不再是独立尾项：其服务端基建被 P1 吸收，排行榜本体排到 P4 之后按反馈上。
- 阶段三（宠物成为界面、大众化）不变；平台的账号与关系链是它的前置资产（领养计划直接受益）。
- 商业模式不变（免费核心 + 化妆品 → Pro 订阅）：Pro 的跨设备 / 舰队监控天然依赖本平台账号体系。
- 桌宠既有用户是手机平台的第一批种子——不推倒任何东西。
