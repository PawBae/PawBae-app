# PawBae Roadmap 人话版

## 一句话

PawBae 是一只陪你写代码的桌面宠物。它不只是可爱，也会看懂你的 AI agents 在忙什么，在你被任务、提醒、token 和权限弹窗淹没时，用更轻松的方式帮你守住工作状态。

## 为什么要做

现在大家都在做“更强的 AI agent”，但真正累的是开发者本人：

- agent 一直在跑，你要不停看它有没有卡住。
- 它做完了、失败了、要权限了，你都要被打断。
- 同时开几个 Claude Code、Cursor、Codex，很容易不知道谁在干什么。
- token 花了多少、是不是浪费了，通常只有事后才知道。
- AI 写了很多代码，但你还得审、改、判断，压力并没有消失。

PawBae 想解决的是这个问题：让开发者不用一直盯着各种终端和面板，也能直觉知道现在的工作状态。

## 它和普通桌宠有什么不一样

Bongo Cat 这类桌宠的爽点是：你一敲键盘，它就有反应；你挂着它，它就陪着你。

PawBae 也要有这种轻松陪伴感，但它更偏开发者：

- 你打字，它会跟着动。
- agent 完成任务，它会给你反馈。
- agent 卡住或失败，它会表现出焦虑。
- token 花太多，它会像“饿了”一样提醒你。
- 你专注写代码时，它尽量不打扰你。
- 你停下来时，它再告诉你刚才发生了什么。

换句话说，PawBae 不是“多一个面板”，而是把复杂的 agent 状态变成一只宠物的表情和动作。

## 现在已经有的基础

这个项目已经不是空想，代码里已经有不少底座：

- 桌面小宠物窗口，可以置顶、展开、拖动、抛掷。
- Claude Code、Codex、Cursor 的 agent 监控。
- 能知道 agent 是在工作、等待、完成，还是有 token 消耗。
- 已经有饥饿度、好感度、金币、喂食、每日奖励。
- 已经能感知键盘和鼠标输入，并让宠物做出反应。
- 已经支持 macOS 和 Windows 的很多桌面差异。

所以下一步不是推倒重来，而是把这些能力串成一个更好玩的体验。

## 第一阶段：先让宠物真的“活起来”

目标：用户一打开 PawBae，就能马上感觉到它不是死的。

### 1. 输入有反馈

你打字，宠物跟着敲；你点鼠标，宠物拍一下。

这件事很重要，因为桌宠最基础的爽点就是“我动，它也动”。它不需要解释，用户马上能感觉到陪伴感。

### 2. 可以用 WASD 控制宠物

用户可以在 PawBae 小窗里用键盘移动宠物：

- `W` 往上
- `A` 往左
- `S` 往下
- `D` 往右
- 方向键也可以
- 按住 `Shift` 可以移动更快

这不是核心生产力功能，但它会让用户觉得宠物是“可玩的”，不是只会站在那里。

后面可以继续做成小游戏，比如带宠物散步、移动一定距离加好感度、长按时播放走路动画。

### 3. 金币要来自真正的开发行为

不要只靠敲键盘刷分。PawBae 的金币应该主要来自更有意义的事情：

- agent 完成任务，奖励多一点。
- 连续专注写代码，奖励中等。
- 输入里程碑，奖励少一点。
- 番茄钟、每日奖励、喂食都进入同一套账本。

这样用户会觉得：我不是在刷无聊积分，而是我的工作节奏真的被看见了。

### 4. 做成就系统

比如：

- 第一次连接 Claude Code。
- 第一次完成 agent 任务。
- 连续专注 30 分钟。
- 累计完成 100 个 agent 任务。
- 连续签到 7 天。

成就不是为了堆功能，而是给用户一个留下来的理由。

## 第二阶段：让宠物看懂 agent

目标：PawBae 开始和普通桌宠拉开差距。

### 1. agent 很忙时，宠物也会紧张

如果你同时跑了好几个 agent，宠物可以表现得有点忙、有点慌。

如果 agent 连续失败、重试、卡住，宠物可以皱眉、冒汗。

如果所有 agent 都空闲，而你在专注写代码，宠物就安静陪着。

用户不需要看一堆状态标签，只要看宠物的表情，就知道现在是不是有点乱。

### 2. token 消耗变成“喂食”

token 账单很抽象，但宠物饥饿度很直观。

可以把 token 消耗理解成宠物吃掉了食物：

- 普通读文件是小零食。
- 大段推理和长上下文是大餐。
- 今天预算快吃完了，宠物提醒你。

这样用户不用打开复杂的成本面板，也能感觉到“今天是不是烧得有点多”。

### 3. 宠物头上显示一句话

agent 做事时，宠物可以冒一个很短的气泡：

- 正在看代码...
- 正在改文件...
- 正在跑测试...
- 需要你看一眼

气泡一定要短，不能变成另一个日志窗口。

## 第三阶段：保护心流

目标：不要让 PawBae 变成新的打扰源。

### 1. 用户专注时，先别吵

如果用户已经连续写了好几分钟代码，PawBae 应该进入“心流保护”。

这时候普通完成提醒先排队，不要立刻弹出来。

等用户停下来，再总结：

> 刚才 3 个 agent 完成了，1 个需要你看看。

这比每完成一个就响一下更适合真实工作。

### 2. 权限提醒分轻重

不是所有权限都应该同样打扰用户。

低风险操作，比如读文件、列目录，可以轻提示。

高风险操作，比如改 `.env`、删除文件、改 shell 配置，才让宠物严肃地提醒。

这块要谨慎做，不能让宠物乱替用户做安全决定。第一版可以先做“分级提醒”，不要急着自动批准。

### 3. 多 agent 总管

当你同时跑多个 agent，PawBae 可以像一个小管家：

- 哪个 agent 在跑，就显示一个小标记。
- 哪个完成了，宠物把结果“叼回来”。
- 哪个卡住了，宠物跑过去提醒你。
- 两个 agent 可能改同一个文件时，宠物表现出冲突警告。

这会让多 agent 工作流更像一个可理解的场景，而不是一堆终端窗口。

## 第四阶段：长期留存

目标：让用户愿意长期挂着 PawBae。

### 1. 饰品和皮肤

宠物可以有帽子、配件、背景小物。

获得方式可以是：

- 金币兑换。
- 成就奖励。
- 定时掉落。
- 特殊开发行为解锁。

但要注意，不要为了收集系统牺牲轻量感。默认体验还是一只安静陪伴的宠物。

### 2. 图鉴

用户可以看到自己解锁了哪些饰品，还有哪些没解锁。

这适合放在面板里，不要直接塞到小窗默认界面。

### 3. 开发者成就墙

PawBae 不应该让用户觉得“代码都是 AI 写的，我只是旁观者”。

它应该强调用户的价值：

- 今天你做了哪些判断。
- 你审了哪些代码。
- 你纠正了 agent 几次。
- 哪些任务是你指挥 agent 完成的。

核心表达是：你不是被 AI 替代的人，你是指挥 AI 的人。

## 第五阶段：发布前打磨

这些不是最炫的功能，但决定产品能不能长期用：

- 游戏模式：全屏打游戏或演示时自动安静，不挡屏幕。
- 开机自启：用户愿意的话，开机自动陪伴。
- 更多动画：打哈欠、伸懒腰、看鼠标、深夜犯困。
- Windows 体验补齐：DPI、窗口位置、全屏隐藏、音频、hook 都要稳定。
- 测试补齐：状态逻辑、奖励逻辑、session 解析、hook 生成都要有测试。
- 多语言：中英文先做好，后面再补日文等。

## 第六阶段：进化和世界（部分已经做完了）

### 1. 宠物会进化（已上线）

你的宠物会随着你真实的工作积累慢慢长大：

- 新生 → 幼苗 → 少年 → 大师 → 传说，五个阶段。
- 经验值就是你赚过的所有金币——agent 完成任务、专注、签到都算。
- 给宠物花钱（喂食）不会扣经验，宠物永远不会退化。
- 从"少年"开始，宠物会根据你的工作风格走出分支：
  - 经常指挥 agent 干活 → **指挥官**（蓝色光环）
  - 长时间专注、爱用番茄钟 → **禅修者**（绿色光环）
  - 喜欢喂食、签到、陪伴 → **贴心伙伴**（粉色光环）
- 进化的瞬间有闪光动画和气泡，跨级也只庆祝一次，不会刷屏。

### 2. 成就和签到（已上线）

- 24 个成就：第一次 agent 任务、十万次输入、相伴一百天……有几个是隐藏的。
- 每日礼物终于有按钮了，连续签到金币越来越多（最多 80），断了也不惩罚。
- 面板会显示"在一起的第 N 天"。

### 3. 网页世界（以后做）

- 先做宠物名片页和 GitHub 徽章，让你可以炫耀。
- 再做星露谷风格的小镇，大家的宠物在里面闲逛、串门、互相投喂。
- 最后做团队牧场：整个团队的宠物住一起，晨会看一眼就知道谁的 agent 卡了。

### 4. 开发彩蛋（以后做）

测试转绿宠物开香槟、force push 宠物吓到炸毛、凌晨三点给你端咖啡。

## 最推荐的实施顺序

先做最容易让用户感知到的东西：

1. ~~输入反馈和 WASD 移动。~~（已完成）
2. ~~金币来源和成就 MVP。~~（已完成，连进化一起做了）
3. agent 情绪和活动气泡。
4. 心流保护和通知队列。
5. 饰品、图鉴和长期收集。
6. Windows、测试、i18n、开机自启等发布打磨。

不要一开始就做很重的权限自动化，也不要过早做复杂 dashboard。PawBae 的第一感觉必须是：轻、可爱、懂我，而且不烦。

## 最重要的产品原则

PawBae 不是为了让用户多看一个工具。

它的价值是：

- 让开发者少盯几个窗口。
- 少被 agent 打断。
- 更早发现 agent 卡住或烧 token。
- 在 AI 写代码越来越多的时候，让开发者仍然感到自己是主导者。

如果一个功能让用户更累，就不要做。

如果一个功能能把复杂状态变成一个自然的宠物动作，就值得做。
